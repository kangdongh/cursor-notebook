# VDB 가속 전략 검토 (2026-03-26)

**작성**: Composer (Cloud Agent)  
**목적**: 벡터 데이터베이스(VDB) 성능·비용을 개선하기 위한 가속 전략을 정리하고, 적용 시 검토 포인트를 명시한다.

---

## 1. 범위와 전제

- **대상**: 임베딩 기반 근사 최근접 이웃(ANN) 검색, RAG 파이프라인의 검색 계층, 대규모 벡터 인덱스 운영.
- **비대상**: 임베딩 모델 자체의 학습 가속(별도 주제).

가속은 보통 **지연 시간(latency)**, **처리량(throughput)**, **비용($/쿼리 또는 $/GB·월)** 중 무엇을 우선할지에 따라 선택이 달라진다.

---

## 2. 인덱스·알고리즘 계층

### 2.1 그래프 기반 (예: HNSW)

| 장점 | 단점 |
|------|------|
| 높은 recall–latency 균형 | 메모리 사용량 큼, 빌드·갱신 비용 |
| 쿼리 경로가 비교적 예측 가능 | 대규모 삽입 시 튜닝·운영 부담 |

**검토 포인트**: `M`, `efConstruction`, `efSearch` 등 파라미터는 recall 목표와 SLA에 맞춰 프로파일링할 것. 메모리 한계에 걸리면 양자화·계층적 인덱스와 조합이 일반적이다.

### 2.2 클러스터링 + 역파일 (예: IVF)

| 장점 | 단점 |
|------|------|
| 메모리 효율·확장에 유리한 경우 많음 | `nlist`·프로브 수에 따라 recall 변동 큼 |
| 파티셔닝과 잘 맞음 | 핫 클러스터에서 병목 가능 |

**검토 포인트**: 데이터 분포가 치우치면 특정 클러스터에 쿼리가 몰려 지연 편차가 커진다. 모니터링 시 **P99**와 클러스터별 QPS를 함께 본다.

### 2.3 양자화 (PQ, SQ, 제품 양자화 변형)

| 장점 | 단점 |
|------|------|
| 메모리·네트워크 대역폭 절감 | recall 손실, 재랭킹 필요 여부 증가 |

**검토 포인트**: **ADC(비대칭 거리 계산)** vs **재구성 벡터** 트레이드오프, RAG에서는 1차 ANN 후 **재랭킹(원 벡터 또는 교차 인코더)** 파이프라인과의 정합성을 명시적으로 설계한다.

### 2.4 하이브리드·필터링

메타데이터 필터(테넌트, 시간, 권한)와 ANN을 동시에 쓸 때, **필터 후 ANN** vs **ANN 후 필터** vs **네이티브 필터 인덱스**의 조합에 따라 tail latency와 정확도가 크게 달라진다.

**검토 포인트**: 필터 선택도가 높은 워크로드는 인덱스 설계를 쿼리 패턴에 맞게 재검토하는 것이 가속보다 효과가 큰 경우가 많다.

---

## 3. 하드웨어·런타임 가속

### 3.1 SIMD / 벡터화

CPU에서 거리 연산 루프의 벡터화는 구현체에 따라 이미 포함되어 있을 수 있다. 자체 커널을 쓴다면 **정렬·패딩·배치 크기**가 캐시 친화적인지 확인한다.

### 3.2 GPU

| 적합 | 부적합 |
|------|--------|
| 대배치 검색, 오프라인 인덱스 빌드, 높은 처리량 | 소배치·초저지연 단일 쿼리(PCIe·커널 런치 오버헤드) |

**검토 포인트**: GPU 메모리 용량 vs 인덱스 크기, **호스트↔디바이스 전송**이 병목이 되지 않는지, 멀티테넌시 시 **배치 합류(batch coalescing)** 정책.

---

## 4. 시스템·데이터 계층

### 4.1 캐싱

- **쿼리/임베딩 캐시**: 동일 또는 유사 질의가 많을 때 효과 큼.
- **핫 세그먼트**: 자주 읽히는 파티션을 빠른 저장 계층에 둔다.

**검토 포인트**: 캐시 일관성(삽입·삭제·TTL)과 캐시 미스 시 tail latency.

### 4.2 샤딩과 복제

- **샤딩 키**: 테넌트·지역·도메인 등으로 분할해 단일 노드 메모리 한계를 넘긴다.
- **읽기 복제**: 읽기 확장; 쓰기 일관성 요구사항에 따라 지연 허용 여부를 정의한다.

### 4.3 압축·스토리지 형식

디스크 기반 세그먼트나 원격 객체 스토리지를 쓰는 경우, **압축률 vs 디코딩 CPU** 균형이 전체 지연에 기여한다.

---

## 5. RAG 맥락에서의 “가속” 재정의

순수 ANN 지연만 줄여도 end-to-end 체감이 개선되지 않을 수 있다.

- **청크 크기·오버랩**: 검색 단위와 후속 LLM 토큰 비용이 함께 움직인다.
- **다단계 검색**: 소형 빠른 인덱스 → 정밀 재랭킹으로 **예산 내 recall**을 맞추는 패턴이 흔하다.
- **평가**: 가속 변경은 **동일 eval 세트**에서 recall@k, MRR, nDCG(가능 시)와 **P95/P99 latency**를 함께 기록한다.

---

## 6. 권장 의사결정 순서

1. **SLA·SLO 정의**: 목표 latency(P95/P99), 목표 recall, 월간 예산.
2. **프로파일링**: CPU/메모리/네트워크/디스크 중 병목 확인(추측으로 인덱스 종류만 바꾸지 않음).
3. **가장 싼 개선**: 필터·쿼리 패턴 정리, 캐시, 파라미터 튜닝, 핫 데이터 배치.
4. **구조 변경**: 양자화, IVF/HNSW 전환 또는 하이브리드, 샤딩.
5. **하드웨어**: GPU 또는 고메모리 인스턴스 등(워크로드가 근거할 때만).

---

## 7. 리스크·운영 메모

- **인덱스 재구축**: 대규모 재빌드는 가용성·디스크·백업 계획과 연동해 롤링 전략을 둔다.
- **버전 호환**: 임베딩 모델·차원·정규화 방식이 바뀌면 인덱스는 **전면 재적재**가 필요할 수 있다.
- **관측 가능성**: 검색 단계별 span(ANN, 필터, 재랭킹, 집계)으로 분해해 병목을 지속적으로 본다.

---

## 8. 요약

VDB 가속은 단일 “최고의 인덱스”가 아니라, **워크로드 병목 + recall 예산 + 비용**의 삼각 균형 문제다. 알고리즘(그래프/IVF/양자화), 시스템(샤딩·캐시·스토리지), 하드웨어(SIMD/GPU)를 순서 있게 적용하고, RAG에서는 **다단계 검색과 평가 지표**를 가속 결정과 같은 테이블에 두는 것이 안전하다.
